手機版響應式優化 — 方案比較

三種實作策略，各有不同的改動範圍與效果深度

A

🔧 由外而內：Shell 優先

先修框架，再修頁面

第一步改好 App Shell（側邊欄→抽屜、Header 適配、觸控尺寸全域放大），建立手機可用的基礎導航。第二步逐頁處理表格和表單。

✅ 優點

  • 第一步做完就能在手機上導航所有頁面
  • 全域 CSS 變數統一調整觸控尺寸
  • 風險最低，每步都可獨立部署

⚠️ 代價

  • 個別頁面（報價編輯器、表格）仍需逐一處理
  • 完整優化需要較多輪迭代

改動範圍：globals.css、Sidebar.tsx、HeaderUserMenu.tsx、button.tsx、input.tsx
預估檔案數：~8 個核心檔 + 各頁面逐步跟進

B

📱 由內而外：核心功能優先

先修報價編輯器（主功能），再擴展到其他頁面

先把使用頻率最高的報價編輯器改成手機友好（品項改為卡片式、輸入框放大），同時順便修導航。其他頁面之後再處理。

✅ 優點

  • 最核心的功能最先可用
  • 用戶感受最明顯的改善
  • 可以先收集回饋再調整其他頁面

⚠️ 代價

  • QuoteEditor 改動量大（500+ 行元件）
  • 其他頁面暫時仍是桌面版體驗
  • 可能需要重構表格結構

改動範圍：QuoteEditor.tsx、Sidebar.tsx、globals.css
預估檔案數：~5 個檔案深度改動

C

⚡ 全面推進：一次到位

Shell + 所有主要頁面一輪做完

結合 A 和 B：先建立 Shell 響應式基礎，接著同時處理所有主要頁面（報價編輯器、報價列表、採購單、案件管理、材料庫）。

✅ 優點

  • 一次完成，體驗一致
  • 不會有「這頁好用那頁難用」的落差
  • 共用元件只需改一次

⚠️ 代價

  • 改動範圍最大（15+ 檔案）
  • 一次 PR 較大，review 負擔重
  • 若有 bug 影響範圍較廣

改動範圍：全部主要元件與頁面
預估檔案數：15-20 個檔案

💡 我的推薦：方案 A（Shell 優先）

原因：目前連基本導航都無法在手機使用。先修好 Shell 層，所有頁面立刻獲得「至少可用」的手機體驗（能導航、能捲動、能點擊）。接著再逐頁深度優化，每一步都能獨立部署和驗證。

方案 C 雖然理想，但改動太集中，風險較高。方案 A 的漸進式路徑最終也能達到方案 C 的效果，但每一步都更安全。
